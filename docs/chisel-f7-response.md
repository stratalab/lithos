## 1. The six decisions

### #1 — Teacher-model doctrine → **ratified: never train on closed-model outputs.**

Two reasons, both decisive: it is **against the closed providers' terms of service** (a
legal/compliance hard line, not a preference), and it is **unnecessary** — the open-weight field
is deep (GLM, DeepSeek, Qwen, Gemma, Nemotron, …) and served cheaply by many cloud inference
providers, so we can collect any teacher outputs we need without ever touching a closed model.

Permitted generators for any training target:

- **Native / mined** content (book exercises, worked examples, solved problems) — G1/G4's core.
- **Open-weight teachers** — generate new SFT solutions, TIR traces, and off-policy pref
  ingredients from any license-permitting open model (GLM / DeepSeek / Qwen / Gemma / Nemotron /
  …), run on a **cloud inference provider or self-hosted**; always **verifier-gated** (the
  "verified synthetic" tier). Harvested open-weight traces (R1 / QwQ / Qwen3, OpenMathReasoning)
  are the ready-made case of this.
- **Lithos self-generation** on the H100 — required specifically for **on-policy** prefs (sample
  the policy → E1-verify); also usable as a teacher.
- **Tool-computed** derivations via the shared verifier (G3) — deterministic, not model output.

**Closed models (Claude / GPT / Gemini-API / …) are build-tools only** — extract, verify, pair,
classify, translate — **never** the source of a training target. This supersedes the proposal's
a/b/c options (it is broader than "self-generation only": *any* open-weight teacher qualifies).
**Label the generating model + provenance on every generated record** so the line is auditable.

### #2 — Verifier seam → **import the Lithos verifier as a package; pin the runtime too.**

The verifier is importable pure functions with no service infra (subprocess + POSIX rlimits,
fully portable), so **import it** — that is "literally your checker," tighter than a service
(which can drift):

```python
from lithos.posttrain.taskbank import Task, load_tasks, verify, verify_batch   # dispatch by kind
from lithos.posttrain.verifier import check_numeric, check_symbolic, check_code, check_units, CheckResult
from lithos.posttrain.sandbox  import run_python, run_octave                    # for G3 key computation
```

`verify(response, task) -> CheckResult` routes by `task.kind`. For **G3** (compute keys), run the
reference solution in the sandbox to derive `answer`; for **G5** (reject bad traces), `verify`
the trace's final answer. Two requirements make the shared-checker guarantee real:

1. **Pin the exact `lithos` version** whose verifier will score the RL run; both repos run the
   same golden-fixture test so drift is caught.
2. **Pin the runtime env, not just the code.** `check_code` executes with the interpreter's
   `sys.executable` + installed libs, so a key you compute in a mismatched env won't match the
   key GRPO checks. Run the verifier in an env matching Lithos's GRPO runtime (Python version +
   the allowed import set in §2 below). If matching the env is painful, fall back to handing
   candidate batches to a Lithos job — but import + pinned-env is simpler; start there.

### #3 — Code-harness contract → **fully pinned by the sandbox** (exact spec in §3 below).

### #4 — TIR spec → **frozen; the producer schema + shared validator are now landed** (§2 below).

### #5 — Split semantics → **Lithos owns the split; stamp `year` + `family_id`; do not pre-split.**

The contamination split is an eval-plan decision (Lithos). **Landed:** `taskbank.split_by_year`
is now **family-aware** — any `family_id` with a post-cutoff member goes *wholly* to eval (so a
pre-cutoff near-duplicate can't leak into training), and `assert_disjoint` now rejects a family
that straddles the split. From Chisel: stamp **`year`** (decontam) and **`family_id`**
(near-dup lineage) on every task; both are first-class fields now (schema §3.6 of the contract).

### #6 — Prefs scope → **confirmed, gated by #1.**

Harvested/synthetic pairs are **off-policy ingredients only**; on-policy prefs stay ours on the
H100 (sample → E1-verify). Schema `{prompt:[{role,content}], chosen, rejected}`
(`preference_dataset`). The synthetic *generators* obey #1: `chosen` mined or generated by an
**open-weight teacher** and verified, never closed-model; `rejected` may be a perturbation or a
weaker open model's output.

## 2. What Lithos just built (the three prerequisites — done + green)

| Fix | Interface Chisel builds against |
|---|---|
| **Shared TIR ingestion validator** | `from lithos.posttrain.tir_validate import validate_tir_record` — call on every SFT/TIR record before emitting; raises `ValueError` with a precise reason. Same function Lithos calls before rendering, so the two can't drift. |
| **Authoritative TIR producer schema** | `docs/tir-format.md` **§7** (records) + **§2–§4** (wire/masking). The old contract "§5" pointer was wrong — §5 is cross-family compat. |
| **Golden fixtures** | `tests/fixtures/tir_golden.jsonl` — the shared canonical set; both repos test that they validate + render. Mirror this file on your side. |
| **Family-aware split** | `taskbank.Task.family_id` + family-aware `split_by_year` / `assert_disjoint`. |

The TIR segment schema (every field a **string**; exactly one of `content`/`segments`;
`segments` assistant-only):

```jsonc
{"type":"think","text":str}                              // learned
{"type":"text","text":str}                               // learned
{"type":"tool","runtime":"python"|"octave","code":str}   // learned
{"type":"tool_result","output":str}                      // MASKED — Lithos appends + masks its <|end|>
```

A `tool_result` answers the immediately preceding `tool` call — emit in call→result order.
Rationale for the loud gate: the `tool_result` span is masked from the loss **by token ID**, so
a mistyped/misplaced segment would silently train the model on the sandbox's output — invisible
until it degrades the model. Validate at both ends.

## 3. The `kind=code` harness contract (for G2), exactly

Pinned by `verifier.check_code` → `sandbox.run_python`:

- **Language: Python (CPython).** Octave exists for TIR *tool calls* but RLVR `kind=code` is
  Python-only — flag if you need Octave code tasks (new work).
- **One shared namespace.** Lithos runs `solution + "\n" + tests` as a single script; your
  `tests` reference the solution's top-level names **directly** (no import dance).
- **Pass = process exits 0.** Use `assert`s; any exception → fail (stderr tail becomes the
  failure detail, and during TIR is injected back as the tool result for retry).
- **Timeout 5.0 s** wall-clock (+1 s CPU backstop). **2 GB** address space.
- **Env:** `{PATH, PYTHONHASHSEED=0, OPENBLAS_NUM_THREADS=1}` — deterministic hashing,
  single-threaded BLAS. **Process-isolation only — no network / filesystem jail** (the model's
  solution is adversarial-for-reward, not assumed adversarial-for-exploit). So tests must not
  touch network/fs and must be deterministic (seed any randomness). Output truncated at 8192 chars.
- **Allowed imports = the GRPO runtime's libs** — stdlib + numpy/scipy/sympy today. **Pin the
  exact set with us before authoring code tests** so a test can't import what the runner lacks.

**Two honest gaps to design around:**
- `check_units` verifies **magnitude only** (not the response's own dimension, Pa vs kPa), and
  `pint` isn't installed yet — so `kind=units` is a value check for now; full dimensional
  checking is a tracked TODO. Prefer `kind=numeric`/`symbolic`/`code` where you can.
- `pint` / `CoolProp` / `python-control` are TIR *tool* libraries; don't assume them in the
  *checker* env until the allowed-import set (above) says so.
